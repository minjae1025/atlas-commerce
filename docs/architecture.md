# Atlas Commerce — Architecture Overview

Atlas Commerce is a B2B wholesale commerce platform. Buyers (retail businesses)
browse a catalog, place bulk orders, and pay through the platform. The system
handles pricing rules, multi-warehouse inventory, order orchestration, payment
processing with an external PSP, and daily financial settlement.

## Services

| Service | Port | Responsibility |
|---|---|---|
| `gateway` | 8080 (exposed) | API gateway / BFF: auth (API keys), routing, rate limiting, response aggregation |
| `catalog` | 7002 | Products, categories, price rules, computed pricing |
| `inventory` | 7003 | Warehouses, stock levels, reservations, adjustments, movements |
| `orders` | 7004 | Order lifecycle orchestration, shipments, cancellation |
| `payments` | 7005 | Payment intents, capture with PSP, FX conversion |
| `settlement` | 7006 | Ledger, daily reports, settlement runs |
| `synthetic-monitor` | — | Continuous synthetic traffic + anomaly alerting (ops tool) |

Infrastructure: PostgreSQL 16 (one instance, schema per service), Redis 7
(caching, idempotency, rate limiting).

## Communication

- All external traffic enters through `gateway` (`/api/<service>/...`).
- Service-to-service calls are direct HTTP (JSON) using the shared
  `serviceClient` from `@atlas/shared`. No message broker.
- Each service owns its database schema exclusively. Cross-service data access
  goes through the owning service's API, never through the database.

## Order flow (happy path)

1. `POST /api/orders/orders` → `orders` validates the customer and items.
2. `orders` → `catalog` `GET /products/:id/price` for each line (price rules applied).
3. `orders` → `inventory` `POST /reservations` (reserve stock, may split across warehouses).
4. `orders` → `payments` `POST /intents` then `POST /intents/:id/capture`.
5. On capture success: `orders` → `inventory` `POST /reservations/:id/commit`
   (decrements on-hand stock) and the order moves to `confirmed`.
6. `payments` and `orders` post ledger entries to `settlement`.
7. On any failure: reservations are released, the intent is voided, the order
   moves to `failed`.

## Runtime

- Node.js 22, TypeScript executed via `tsx` (no build step).
- Each service is an Express app created by `createApp()` in `src/app.ts` and
  bootstrapped by `src/index.ts`.
- `docker compose up` starts everything: infra, migrations/seed (one-shot
  `migrator` container), all services, then `synthetic-monitor`.
- All base images are official multi-arch images. Never set `platform:` in
  compose or Dockerfiles.

## Repository layout

```
packages/shared/        @atlas/shared — logger, config, db, cache, http, retry,
                        idempotency, errors, middleware, money
services/<name>/        one directory per service (see docs/contracts.md)
db/migrations/          numbered SQL migrations (run by migrator container)
db/seed/                seed SQL + data
docs/                   this documentation
docker-compose.yml
```

## Conventions

- Money is always integer minor units (`*_cents` columns, `amountCents` fields).
- Timestamps are UTC ISO-8601 strings in APIs, `timestamptz` in PostgreSQL.
- Logs are structured JSON lines via the shared logger (`service`, `level`,
  `msg`, `requestId`, plus context fields).
- Every service exposes `GET /health` (liveness) and `GET /ready` (checks DB +
  Redis + downstream dependencies it needs at boot).
- Errors use the shared `AppError` hierarchy; the shared `errorHandler`
  middleware maps them to JSON `{ error: { code, message } }` responses.
- IDs are UUIDv4 generated by the owning service.
